# `design-review-lite`（轻量设计审查）

## 作用

用这项插件做一轮**轻量产品 / 设计 快速合理性检查**。

它不是完整设计系统评审，也不是高保真 UX 流程；它只是一个快速发现明显交互、清晰度、流程与状态问题的方法，避免实现一路跑偏。

## 适用场景

适合：
- UI 页面
- 多步骤用户流程
- 设置页 / dashboard 的信息架构
- 表单与校验体验
- 功能逻辑没错，但可能让用户 / 操作者迷惑的界面
- 功能上说得通，但用起来可能别扭的产品 范围

## 输出合约

输出一份紧凑的评审结果，包含：
- 核心用户任务
- 主要清晰度风险
- 流程摩擦点
- 缺失状态 / 边界情况
- 推荐简化项
- 设计 判定

## 审查视角

重点看：
- 用户目标是否清晰
- 是否存在多余步骤
- 标签或结构是否让人误解
- 是否缺 empty / loading / error 等状态
- v1 是否暴露了过多选项
- 业务逻辑与用户心智模型是否错位

## 默认 判定

可用这些判定：
- 够用了，可以直接 build
- 可以 build，但先做简化
- 先修流程，再 build
- 卡在产品决策上，暂时 blocked

## 工作流

### 步骤 1 — 重述用户任务

先说清楚用户 / 操作者到底想完成什么。

### 步骤 2 — 走一遍主流程

要问：
- 达成目标的最短路径是什么？
- 用户会在哪一步犹豫或误解？
- 哪些地方明显比必要更重？

### 步骤 3 — 检查状态覆盖

重点找缺失的：
- empty states
- loading states
- validation states
- failure states
- permission / blocked states

### 步骤 4 — 激进简化

默认偏向更少：
- 控件
- 分支
- 字段
- 页面
- 选择项

### 步骤 5 — 给出设计 判定

最后必须给 main 编排器一个**能直接拿去用**的设计建议。

## 交接建议

适合在这些位置插入：
- UI-heavy 工作的 coding 之前
- `founder-office-hours` 之后，如果产品 framing 已清楚，但界面形状还虚
- `qa-browser-check` 之前，如果需要先做一轮结构 sanity pass

## 示例触发

- “先帮我快速过一下这个页面 / 流程设计”
- “这个功能逻辑没问题，但交互可能还不顺”
- “别上完整设计流程，先做一轮轻量设计审查”
