# `review-gate`（质量门）

## 作用

把松散的“帮我 review 一下”变成一个清晰的**能不能继续往下走**的质量门。

重点不是风格评论，而是判断：
- 这次改动到底对不对
- 是否足够完整
- 是否足够安全继续推进
- 下一步必须是什么

## 适用场景

适合：
- 已完成的 coding 任务
- 工作流 / 自动化变更
- 带运营风险的配置修改
- 在 QA / browser check 前的 UI 或产品改动
- 任何“看起来像做完了”但实际上不能只凭感觉判断的任务

不适合：
- 代码还没开始前的 brainstorming
- 只改了错别字的 trivial 修改
- 只需要校对文字的任务

## 输出合约

输出一份简洁的质量门结果，包含：
- Verdict（判定）
- 审查了什么
- 正确性担忧
- 回归风险
- 完整性缺口
- 必需测试 / 验证
- 推荐下一步

允许判定：
- Pass
- 通过，但有后续项
- QA 前需先修复
- Blocked

## 审查标准

重点看：
- 正确性
- 回归风险
- 隐藏副作用
- 缺失边界情况
- 请求范围与交付范围是否错位
- 验证与测试覆盖是否不足
- 上线 / 运行风险

不要为了礼貌牺牲判断。

## 工作流

### 步骤 1 — 先重述目标结果

说明这次改动声称解决什么问题，什么才算成功。

### 步骤 2 — 检查正确性

要问：
- 它真的解决了目标问题吗？
- 有没有逻辑漏洞或错误假设？
- 是否违反已有约束或架构？

### 步骤 3 — 检查回归风险

要问：
- 哪些相邻行为可能被打坏？
- 哪些现有流程依赖这些文件或决策？
- 哪些点最脆弱、最容易漏？

### 步骤 4 — 检查完整性

要问：
- 有没有必要的后续没做？
- docs、tests、migrations、prompts、configs 是否不同步？
- 有没有“部分完成却被报成完成”？

### 步骤 5 — 给出质量门判定

必须给一个明确 判定。

如果可以进入 QA，要明确 QA 该重点验证什么。
如果必须回到 coding，要明确必须先修什么。

## 交接规则

往下交接时：
- 给 QA：写清楚最高风险用户路径 + 最可能失败模式
- 给 Docs：写清楚这次改变了什么、用户 / 操作者现在要知道什么
- 退回 Coding：给出最小必要修复清单，说明什么改完就能改变 判定

## 示例触发

- “帮我过一遍这个改动，看能不能进下一步”
- “别只看 diff，帮我做一次真正的 review gate”
- “这个功能到底算不算完成？”
- “先 review，再决定要不要做 browser QA”
