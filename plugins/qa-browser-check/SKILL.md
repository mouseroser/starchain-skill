# `qa-browser-check`（浏览器 QA）

## 作用

把“去浏览器里测一下”变成一轮**有目标流程、有预期结果、有失败证据**的聚焦 QA。

目标不是广泛乱点，而是：
- 验证关键路径
- 抓关键回归
- 返回清晰的失败证据与下一步建议

## 适用场景

适合：
- UI 改动
- 多步骤工作流
- 浏览器可见的功能交付
- dashboard、表单、设置页
- 发布 / 创作者工作流
- 依赖登录态、本机浏览器状态的检查

不适合：
- 完全无浏览器可见行为的后端改动
- 没有可运行路径的概念评审
- 没有目标路径定义的宽泛 exploratory testing

## 输出合约

输出一份 QA 说明，包含：
- 目标流程
- 使用的环境 / 浏览器路径
- 结果
- 证据
- 发现的问题
- 推荐下一步

允许结果：
- Pass
- 通过，但有注意事项
- Fail
- 被环境阻塞

## 测试原则

宁可验证**一条强而真实的主路径**，也不要做很多浅层点击。

这轮检查应该回答：
- 目标用户能不能完成关键流程？
- 实际行为是否符合已审过的计划？
- 具体在哪一步坏掉？
- 构建者最应该先修哪里？

## 工作流

### 步骤 1 — 先定义目标流程

动手点之前，先写清楚：
- 起始状态
- 用户目标
- 关键路径步骤
- 预期结果

如果没有清晰目标流程，就要求补充，或者推断业务上最关键的一条路径。

### 步骤 2 — 选对浏览器上下文

优先使用现有 OpenClaw 浏览器栈。

如果 cookies、会话、创作者平台权限很重要，优先使用**已登录用户浏览器 / profile**。

没有充分理由，不要随意切换浏览器栈。

### 步骤 3 — 验证主路径

跑最短但真实的一段序列，用来证明或证伪功能。

记录：
- 从哪开始
- 关键交互点
- 看到的结果或失败

### 步骤 4 — 检查最可能的回归边

主路径之后，只补测**一个最可能的相邻失败点**：
- 表单校验边界
- 状态持久化问题
- 错误跳转
- 漏掉 save / apply
- modal / dialog / permission 断裂

### 步骤 5 — 清晰汇报

如果通过，要说清楚通过了什么。

如果失败，要明确：
- 失败的精确步骤
- 预期行为 vs 实际行为
- 可见错误或破损状态
- 如果明显，给出修复方向

## 缺陷汇报规则

一条好的 QA bug 报告至少包含：
- 页面 / 流程名
- 失败的精确步骤
- 预期行为
- 实际行为
- 对目标流程的严重性

避免“感觉坏了”这种模糊说法。

## 示例触发

- “帮我做一次 browser QA”
- “这个 UI 改动过 review 了，再跑一遍关键路径”
- “发布流程改了，帮我验证主链有没有断”
- “先在浏览器里走一遍，再决定能不能 release”
