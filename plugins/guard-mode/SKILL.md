# `guard-mode`（局部护栏）

## 作用

这项插件用于给一条 StarChain 任务临时加上**执行边界**。

目标不是替代系统级安全规则，而是增加任务局部护栏，让运行始终留在允许的爆炸半径内。

## 适用场景

当任务需要这些约束之一时使用：
- 只允许改某个目录
- 只读审查
- 不允许碰配置
- 不允许碰部署 / runtime 文件
- 危险命令先预警
- 将 范围 冻在一个小子系统
- 生产排障中只允许观察，不允许随意变更

## 输出合约

输出一份简短 护栏配置，包含：
- 允许范围
- 禁止范围
- 编辑策略
- 命令策略
- 升级触发条件
- 退出条件

## 常见护栏模板

### 目录绑定编辑模式
当只允许修改某一 app / module / path 时使用。

需要定义：
- 允许目录
- 禁止目录
- 是否允许新建文件
- 是否允许 rename / delete

### 只读审查模式
当任务只允许分析 / review 时使用。

需要定义：
- 不允许文件编辑
- 不允许 write 命令
- 不允许 config 变更
- 输出只能是 findings

### 配置冻结模式
当允许改代码，但不允许改配置时使用。

需要定义：
- config 路径冻结
- schema / runtime 设置冻结
- env / secret / config 文件禁改

### 危险动作预警模式
当清理、迁移、重置、删除等动作可能进入范围时使用。

需要定义：
- 危险动作前先提醒
- 优先非破坏性检查
- 在真正 destructive 之前必须有显式 go / no-go 检查点

## 默认策略写法

护栏语言要写得**短、清楚、可执行**：
- 只允许修改 `X`
- 不要改 `Y`
- 把 `Z` 当成只读
- delete / reset / migrate 之前先预警
- 如果任务膨胀到允许范围外，立即停止并升级

## 升级规则

满足以下情况要升级或暂停：
- 根因在允许范围外
- 要完成任务必须修改禁止目录
- 任务从审查变成实际变更
- 必须做危险动作
- 请求本身存在相互矛盾的边界

## 交接规则

在 StarChain 内使用时：
- 在 coding 或 root-cause investigation 开始前先套上 `guard-mode`
- 把 护栏配置 直接写进 spawned task prompt
- profile 要短而具体
- 如果 profile 阻止了完成，就返回最小必要的 范围 扩张请求

## 示例触发

- “这次只准改 apps/web”
- “先只读审查，不要动代码”
- “不要碰配置和部署，只看业务逻辑”
- “这次排障先观察，危险命令先提示”
